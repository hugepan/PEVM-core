# ONTLG 工程复盘：三个根因 Bug

**角色**: 041·ONTLOGY 对标 Palantir·FDE 三层应用闭环
**模型**: ds_flash

---

问：你发现的三个根因 bug 是什么？它们之间有什么共同模式？

三个根因 bug，按我发现它们的顺序说。

**第一个：原始素材和精炼结论没有分离。** 我当时以为它们只是一个属性标签，写在哪都行。结果做全面检查的时候发现——原始素材文件里混着精炼结论的声明，精炼目录里躺着原始素材级别的半成品。整个数据摄入层在对着错误的源做清洗。这不是拼写错误，是概念封装没到位——一个文件不知道自己是"原始素材"还是"精炼结论"，那下游所有依赖它的层都在猜。

**第二个：决策终止机制缺失。** 我在做对标的时候，反复陷入"这个映射还不够细，那个特性还没覆盖"的循环。进度条卡在 90% 永远到不了 100%。这不是技术问题，是决策机制缺失——没有一条明确规则说"证据链完整、闭环跑通、该停就停"。结果时间耗费在无限精修上，而不是验证核心假设。

**第三个：上下文噪音污染。** 对话历史的上下文里混入了大量不相关的调试话、废弃的思路、甚至之前失败的实验数据。当我调用系统时，它把噪音当作有效信号喂给我，导致后续推理被带偏。这是输入质量控制问题——你想让系统聪明，得先保证它吃进去的是干净数据。

**共同模式：边界不清晰。** 三者都是"什么属于什么"的界定问题——原始素材和精炼结论之间的物质边界、进展与完美之间的时间边界、有效信号与噪音之间的质量边界。每个 bug 本质都是"某条分界线画错了或者没画"。

如果只留一个，我选决策终止机制。因为它是一个元规则——其他两个 bug 修复后，如果没有明确的"够了"信号，你会在"继续修到完美"的循环里耗尽精力，永远不会宣布"当前版本可用"。素材与结论的分离可以慢点修，上下文清理可以分批做，但终止机制不装好，整个系统的节奏就是失控的。它像扳道工——不装它，火车永远在线路上绕圈圈。
