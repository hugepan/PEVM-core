# PEVM 架构总览

> 认知协议栈——在人脑心智模型与 AI 原生表达之间建立翻译层。
> 不是一个框架。是一套数据格式 + 协议栈 + 参考实现。


## 1. PEVM 是什么

PEVM 位于人脑心智模型与 LLM 消费之间：

```
人脑心智模型 → [PEVM 协议层] → LLM 消费
```

PEVM 的核心由两部分构成：

1. **结构化卡片格式**——将任意实体（人物、概念、公司、物体）编码为 ~3000 token 的结构化数据，任何 LLM 都能直接消费。每张卡携带认知层级指标（`register 1-10`）和三维人格基调（`root_loss`），无需提示词工程即可实现精细的人格控制。

2. **协议栈**——卡片上的操作：融合（多卡焊接为新实体）、辩论（多卡结构化圆桌）、场景应用（卡片应对真实任务）、时间线（同一个人在不同人生阶段的切片）、渲染（内部格式转换为面向人类的输出）。

核心原则：**好的数据格式 > 好的提示词 > 好的人格**。PEVM 把"让 AI 理解一个概念"从提示词工程问题转变为数据格式问题。


## 2. 四层架构

PEVM 按职责分为四层：

```
┌──────────────────────────────────────────────────────────────┐
│  Layer D: 远景——AI 原生操作系统（探索方向）                    │
│  研究方向：卡片 FUSE 文件系统、LLM 系统守护进程、               │
│  人格边界感知内核钩子                                          │
├──────────────────────────────────────────────────────────────┤
│  Layer C: 系统——实体生命周期管理                               │
│  创建 → 融合 → 分裂 → 时间线 → 删除                           │
│  训练数据工厂 · 系统监控 · CLI 工具链                         │
├──────────────────────────────────────────────────────────────┤
│  Layer B: 协议——卡片上的操作                                   │
│  融合 · 辩论 · 场景应用 · 挂载 · 渲染                        │
├──────────────────────────────────────────────────────────────┤
│  Layer A: 格式——结构化卡片数据                                 │
│  register/root_loss · 结构化叙事 · 安全机制                   │
│  ~3000 token 每卡，任意 LLM 可消费                           │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  LLM 推理层（云端或本地）                                      │
└──────────────────────────────────────────────────────────────┘
```

**关键约束**：下层不知道上层存在。Layer A（数据格式）没有"协议"的概念。Layer B（协议层）不关心后端是哪个 LLM。这种分离保证每层可以独立演进。

### Layer A: 结构化卡片数据

一张卡是一个自包含的 markdown 文件，三段结构：

- **元数据头**——认知层级（register 1-10）、三维人格基调（root_loss）、实体关系、卡间张力信号
- **结构化叙事**——精炼的因果链，在各阶段保留关键信号
- **安全机制**——防止幻觉漂移、强制人格边界的护栏

设计约束：≤3328 token 每卡（结构元素除外）。每张卡必须能在单次上下文窗口中被任何 LLM 消费。

### Layer B: 协议栈

协议是对卡的操作。正如 Unix 对文件有 open/read/write，PEVM 对卡有融合/辩论/应用：

| 协议 | 功能 |
|------|------|
| **融合（焊接）** | 将 N 张卡合并为具有合成人格的新实体 |
| **圆桌辩论** | N 张卡的结构化辩论——逐轮收敛 |
| **场景应用** | 实体应对真实任务（匹配分析、技术尽调、战略咨询） |
| **挂载** | 运行时附加外部资源（原材料、参考文档）到卡 |
| **渲染** | 内部格式转换为面向人类的输出（技术报告、社交媒体、画廊） |

### Layer C: 实体生命周期

卡片通过 CLI 工具管理完整的生命周期：

| 操作 | 描述 | 状态 |
|------|------|:----:|
| 创建 | 从原材料生成新卡 | ✅ 管线 |
| 精修 | 基于参考材料增量改进 | ✅ 已使用 |
| 融合 | 多卡合并为融合体 | ✅ 52 张融合卡 |
| 分裂 | 从主卡生成子人格 | ✅ 已交付 |
| 时间线 | 同一个人在不同人生阶段的切片 | ✅ 9 个时间切片 |
| 上身对话 | 单卡人格模拟 | ✅ 325+ 条记录 |
| 圆桌辩论 | 多卡结构化辩论 | ✅ 多个话题 |
| 审计 | 自检结构完整性 | ✅ 管线 |
| 删除 | 移除卡片 | ✅ 已交付 |

### Layer D: 远景（探索方向）

研究方向——将协议栈嵌入操作系统：

- 卡片文件系统挂载到 Linux VFS
- LLM 作为系统守护进程（内核级推理）
- 人格一致性校验作为 AI 原生看门狗
- 输入校验作为 AI 原生系统日志

非当前声明。面向合作者的研究方向。


## 3. 数据流

```
原材料
    │
    ▼
① 入库：注册占位 → 卡片生成（LLM 产出卡片）→ 写回注册表
    │
    ▼
② 结构化卡片（按域组织）
    │
    ├── 上身对话    → 单卡人格响应
    ├── 圆桌辩论    → 多卡结构化辩论
    ├── 场景应用    → 基于场景的推理
    ├── 融合        → 多卡合并为新实体
    ├── 分裂        → 子人格生成
    ├── 时间线      → 人生阶段切片
    ├── 渲染        → 格式转换面向人类阅读
    └── 删除        → 清理
    │
    ▼
③ 产出：交互记录 / 渲染格式 / 训练数据
```


## 4. 设计原则

1. **下层不知道上层存在**——数据格式没有协议概念。协议不关心后端是哪个 LLM。各层独立可替换。

2. **本地优先**——本地 LLM 回退（消费级 GPU 可用）不是可选功能——是架构决策。一切必须能在无云依赖下工作。

3. **短而密的张力信号优于长描述**——焊接质量取决于卡间精确的张力信号。单行对立（"螺纹 vs 梁——自由 vs 承重"）的一致表现优于多段学术描述。

4. **好的数据格式 > 好的提示词 > 好的人格**——卡片是结构化数据，不是提示词模板。格式本身强制的一致性没有任何提示词能保证。

5. **主权是架构层面的**——不是功能开关，不是配置项。系统通过结构保证隐私：卡片数据存在本地，推理可本地运行，格式是自包含的。

6. **输出约束集中化**——术语规则、数据边界、安全过滤器统一定义并引用，从不重复。


## 5. 开发效率

一个数据点：timeline 功能（同一个人在不同人生阶段的切片）从概念到可用实现花了 **5+5+5 分钟**。不是因为简单——是因为 PEVM 的架构设计使得每种新操作都遵循相同的正交原语。分裂、时间线、融合、组团——每个新协议都建立在已有的基础之上。

这种速度不是偶然。是"下层不知道上层存在"原则的直接结果：加新协议不改数据格式，加新操作不改已有协议。
